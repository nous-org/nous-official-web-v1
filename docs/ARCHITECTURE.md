# Architecture

This document describes the production shape of the NOUS website and the boundaries that should remain stable as the site evolves.

## System Overview

```text
User, crawler, or bot
  -> Cloudflare route: nous.cr/* and www.nous.cr/*
  -> Astro middleware
  -> Astro server-rendered route
  -> selective React island hydration
  -> integrations as needed
```

The site is intentionally hybrid:

- Astro owns the page shell, routing, SEO-critical markup, and most rendered content.
- React is used for interactive islands such as service tabs, animated sections, and admin tooling.
- Cloudflare Workers provide the runtime, edge routing, static asset serving, and KV-backed session support.

## Runtime Boundaries

| Boundary | Owner | Notes |
|---|---|---|
| Routing and pages | `src/pages` | File-based Astro routes plus API endpoints |
| Shared layout | `src/layouts/Layout.astro` | Metadata, schema, favicon, language alternates, header/footer shell |
| Redirects and headers | `src/middleware.ts`, `public/_redirects`, `public/_headers` | Keep canonicalization and legacy URL handling aligned |
| SEO components | `src/components/seo` | Canonical, Open Graph, schema, breadcrumbs, service and article markup |
| Content collections | `src/content`, `src/content/config.ts` | Legal pages and content typed through Astro content |
| Localization | `src/lib/i18n.ts` | Locale normalization, localized paths, shared copy |
| Admin auth | `src/lib/auth.ts` | Clerk verification plus explicit admin policy |
| Database access | `src/lib/db.ts` | Turso/libSQL clients and content data access |
| Contact workflow | `src/pages/api/contact.ts` | Server validation, abuse controls, and Resend e-mail delivery |
| Runtime config | `src/lib/runtime-env.ts` | Environment handling without leaking secrets |
| Deployment | `wrangler.toml`, `.github/workflows/deploy.yml` | Cloudflare Worker config and CI deploy |

## Route Strategy

English content uses canonical root-level routes. Spanish content uses `/es`.

```text
/                  English homepage
/es                Spanish homepage
/services          English services
/es/services       Spanish services
/about             English company page
/es/about          Spanish company page
/contact           English conversion route
/es/contact        Spanish conversion route
/blog              English blog index
/es/blog           Spanish blog index
```

Retired routes are intentionally redirects:

```text
/pricing     -> /services
/products    -> /services
/about-us    -> /about
/contact-us  -> /contact
```

Do not reintroduce source pages for those retired URLs unless the SEO strategy changes.

## Rendering Strategy

Critical SEO content should be available in the initial HTML whenever practical:

- page title
- meta description
- canonical tag
- hreflang alternates
- primary heading
- service/content copy
- internal links
- structured data

React islands should not be required for crawlers to understand the primary page topic or navigation.

## Structured Data

Structured data is emitted through dedicated components:

- `OrganizationSchema.astro`
- `WebSiteSchema.astro`
- `ServicesSchema.astro`
- `ArticleSchema.astro`
- `FaqSchema.astro`
- `BreadCrumbs.astro`

Schema should describe visible content and should use stable `@id` references for the NOUS organization and website entities.

## Sitemap And Indexation

The sitemap is generated by Astro in `astro.config.mjs`. The repo should not contain a manual `public/sitemap.xml`.

Excluded or noindexed surfaces include:

- `/404`
- `/admin`
- `/portfolio`
- retired redirects
- API routes

When new pages are added, decide whether they should be:

- indexable and included in the sitemap
- noindexed but crawlable
- redirected
- private and blocked

## Contact Form And E-mail Flow

The contact form is split between Astro-rendered markup, a small client validation script, and a server-side API endpoint:

- `src/components/ui/ContactForm.astro` and `src/components/ui/ContactPageForm.astro` render the public forms.
- `src/components/ui/ContactFormClientScript.astro` owns inline validation states and submit behavior.
- `src/pages/api/contact.ts` owns authoritative Zod validation, honeypot handling, rate limiting, safe HTML escaping, and Resend delivery.
- `public/images/nous-email-logo.png` is the logo used by the contact e-mail templates.

The endpoint sends an internal notification to `CONTACT_RECIPIENT_EMAIL` and a confirmation e-mail to the submitter. User-submitted values must be escaped before being rendered into HTML e-mail templates.

See [CONTACT_AND_EMAIL.md](CONTACT_AND_EMAIL.md) for template standards and smoke-test details.

## Design System Notes

The site uses a dark, restrained B2B AI-transformation aesthetic. Keep the interface:

- direct and consultative
- quiet rather than hype-driven
- readable on mobile
- clear about business value
- consistent across English and Spanish

Avoid one-off visual treatments that do not connect to the existing palette, typography, spacing, or interaction patterns.
